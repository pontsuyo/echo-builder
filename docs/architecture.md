### 5.2 詳細データフロー: 音声認識

```
1. ユーザーがマイクボタンをクリック
   → VoiceRecognitionComponent.startRecording()

2. ブラウザがマイクアクセス許可をリクエスト
   → navigator.mediaDevices.getUserMedia({ audio: true })

3. 音声データを2秒間隔でキャプチャ
   → MediaRecorder.ondataavailable

4. 音声チャンクをプロキシサーバーに送信
   → POST /api/voxtral/transcribe (streaming)

5. プロキシサーバーがMistral APIに中継
   → POST https://api.mistral.ai/v1/audio/transcriptions

6. 転写結果をストリーミングで受信
   → TextEventStream

7. リアルタイムでUIに表示
   → dispatchEvent('transcript', { text, isFinal })

8. 完了またはエラー時にコールバック実行
   → onComplete() 或いは onError()
```

### 5.3 詳細データフロー: 建築生成

```
1. 音声入力完了イベント受信
   → eventBus.emit('voice:complete', { text })

2. 建築理解エンジンがテキスト解析
   → BuildingUnderstandingEngine.analyze(text)

3. パラメータ抽出
   → extractParameters(tokenizedText)

4. 曖昧理解適用
   → applyAmbiguousUnderstanding(params, workerProfile)

5. 建築パラメータ生成
   → generateBuildingParameters()

6. 描画エンジンにパラメータ渡し
   → RenderingEngine.drawBuilding(params)

7. アニメーション開始
   → AnimationSystem.start(buildingAnimation)

8. 進捗イベント発火
   → eventBus.emit('building:progress', { progress })

9. 建築完了
   → eventBus.emit('building:complete', { building })
```

## 8. 設計決定

### 8.1 主要な設計決定

| 決定 | オプション | 理由 |
|------|----------|------|
| 単一HTMLファイル | マルチページ vs 単一ページ | インストール不要、簡単な配布 |
| バニラJavaScript | フレームワーク vs バニラJS | 軽量、学習コスト低、パフォーマンス |
| Canvas API | SVG vs Canvas | ピクセルアートに適切、パフォーマンス優位 |
| プロキシサーバー | クライアント直 vs プロキシ | APIキーの安全管理、CORS回避 |
| FastAPI | Flask vs FastAPI | 非同期サポート、パフォーマンス |
| Mistral Voxtral | Whisper vs Voxtral | リアルタイムストリーミング対応 |

### 8.2 トレードオフ

#### 8.2.1 単一HTMLファイル

**利点**:
- インストール不要
- 簡単な配布
- オフラインでも動作（音声機能を除く）

**欠点**:
- コードのモジュール化が難しい
- 大規模化に向かない
- ビルドプロセスが簡素

**緩和策**:
- モジュールパターンの採用
- 名前空間の使用
- コード分割（別ファイル読み込み）

#### 8.2.2 バニラJavaScript

**利点**:
- フレームワークの学習コストなし
- パフォーマンス優位
- バンドルサイズが小さい

**欠点**:
- 状態管理が複雑
- コードの組織化が難しい
- テストがしにくい

**緩和策**:
- カスタムイベントバスの実装
- モジュールパターンの採用
- TypeScriptの導入（将来的）

#### 8.2.3 プロキシサーバー

**利点**:
- APIキーの安全管理

**欠点**:

**緩和策**:

- フォールバックメカニズム
