# 開発ガイドライン - Echo Builders

## 1. 開発方針

本プロジェクトは、現時点では最小構成で進めることを優先する。  
優先は `p0` の可動状態維持、`p1` は仕様上は記録しつつ実装を先送りする。

## 2. 優先度（実装方針）

- **p0（必須）**: 音声入力・指示解釈・建築進行
- **p1（検討）**: ElevenLabs/TTS、スコアリング、互換拡張

## 3. 開発環境

- ブラウザ: 開発は任意ブラウザ（主に Chrome/Firefox）
- サーバー: Python 仮想環境（`requirements.txt` のみ）
- 実行: `index.html` を開き、`mistral_proxy.py` を起動
- 音声: ローカルのマイク許可が必要

## 4. 変更手順（小さく・短く）

1. 変更対象ファイルを1つ決める
2. 変更前後の既存仕様への影響を `docs` と比較
3. コード変更後、関連ドキュメント（PRD/FD/architecture）を同時に微修正
4. `R` でゲームをリセットして基本シナリオを手動確認
5. 1機能ずつコミット（広い変更を混ぜない）

## 5. コーディング規約（現行ルール）

### JavaScript
- 2スペースインデント
- `let`/`const` を明示、セミコロンは既存方針に従う
- 命名は `camelCase`
- 深い入れ子回避。必要なら早期 return で短くする
- 文字列比較は厳密一致 (`===`)

### Python
- 4スペースインデント
- 例外は具体的に捕捉
- ログは `print` より `logging` を優先
- HTTP ハンドラでは `HTTPException` / 明示エラーメッセージ

### ファイル分割
- `index.html` は起点のみ
- 音声連携は `voxtral.js`
- コアロジックは `game-commands.js` / `game-engine.js` / `game-data.js`
- 追加 API は `mistral_proxy.py` 側に明確に分離

## 6. 命名とAPI整理ルール

- ゲーム側表現は `Player` を使う。既存の `hero` 系コールバック名は将来整理対象とし、すぐに変えないならコードコメントで注記する。
- p1 から入れる関数名は将来重複しないよう `Player` 文脈で揃える。
- 仕様上の新規フィールドは、既存状態オブジェクトに直接追加せず、まず `game-*` の影響範囲を確認してから反映。

## 7. テスト/確認（実装方針）

### 7.1 必須確認項目（各変更ごと）
- 起動
  - indexを開いて初期表示が崩れない
- 音声UI
  - `M` キーまたはボタンで開始・停止できる
- 解釈
 - 指示入力が `isBuild` 判定される
- 建築
  - NPC に割当が進み、パーツ完成時に結果ログが更新される
- リセット
  - `R` で状態が初期化される

### 7.2 ログ監視
- 主要な状態変化は console でも確認できるよう維持
- 連続エラーが出る場合は `voxtral.js` の再試行条件と fallback を優先して修正

### 7.3 自動化
- 現時点では自動テストを必須化しない（リポジトリ構成上未整備）
- 重要な変更のみ、手動フロー確認を実施

## 8. エラー対応

- マイク許可不足: 画面メッセージを表示し、再試行導線を案内
- 文字起こし失敗: 再試行可能かを提示
- コマンド空振り: `isBuild=false` でも進行を止めず次ターンへ移す
- 例外増殖時: 例外箇所と連動箇所を 1 ファイルずつ切り分ける

## 9. レビュー観点

- p0 機能の壊れ: 音声→解釈→建築の連鎖が成立するか
- 命名の一貫性: `Player` / `NPC` など主要語が仕様と揃っているか
- p1 決定の混入: 未実装項目を現行実装として扱っていないか
- 不要複雑化: 一度に増やした要件に対して分割できるか

## 10. リリース準備（軽量）

- ドキュメント更新日とバージョンを同期（`docs/*`）
- `README.md` と `.env.example` が実行前提と一致しているか
- 既知の未実装機能は明示したままにする
